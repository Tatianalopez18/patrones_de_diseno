PATRONES DE COMPORTAMIENTO

Estos patrones se enfocan en la comunicación entre objetos, organizando las responsabilidades y el flujo de control para lograr un sistema más cohesionado y flexible.

Chain of Responsibility
  • Definición: Permite que una solicitud pase por una cadena de objetos hasta ser atendida.
  • Usos: Validadores, manejo de eventos.
  • Ventajas: Separación de responsabilidades.
  • Desventajas: Difícil saber cuál objeto responderá.
  • Patrones relacionados: Command, Composite.
Command
  • Definición: Encapsula una solicitud como objeto, permitiendo parametrizar acciones y soportar deshacer/rehacer.
  • Ejemplo: Menús y botones de interfaz gráfica.
  • Ventajas: Alta flexibilidad.
  • Desventajas: Puede crear muchas clases.
  • Usos: Sistemas de tareas, historial de acciones.
  • Patrones relacionados: Memento, Invoker.
Interpreter
  • Definición: Interpreta gramáticas y lenguajes definidos por reglas.
  • Usos: Lenguajes específicos de dominio (DSLs).
  • Desventajas: Difícil de mantener con gramáticas extensas.
  • Patrones relacionados: Composite, Visitor.
Iterator
  • Definición: Proporciona una forma de recorrer los elementos de una colección sin exponer su estructura interna.
  • Ventajas: Acceso uniforme.
  • Desventajas: Puede implicar sobrecarga en ciertas implementaciones.
  • Usos: Recorrido de listas, matrices.
  • Patrones relacionados: Composite, Factory Method.
Mediator
  • Definición: Centraliza la comunicación entre varios objetos para evitar dependencias directas.
  • Usos: Interfaces gráficas.
  • Ventajas: Menor acoplamiento.
  • Desventajas: Puede convertirse en una clase muy compleja.
  • Patrones relacionados: Observer, Facade.
Memento
  • Definición: Permite capturar y restaurar el estado de un objeto sin violar su encapsulamiento.
  • Ejemplo: Funciones “deshacer” en editores.
  • Desventajas: Consumo de memoria.
  • Usos: Juegos, editores, pruebas.
  • Patrones relacionados: Command, Prototype.
Observer
  • Definición: Permite que múltiples objetos se suscriban y reaccionen a cambios en otro objeto.
  • Usos: Sistemas de eventos, interfaces reactivas.
  • Desventajas: Puede generar dependencias ocultas.
  • Patrones relacionados: Mediator, Event Dispatcher.
State
  • Definición: Permite que un objeto modifique su o según su estado interno.
  • Ventajas: Mejora organización del código.
  • Desventajas: Aumenta el número de clases.
  • Usos: Máquinas de estados, juegos, workflows.
  • Patrones relacionados: Strategy, Flyweight.
Strategy
  • Definición: Permite definir diferentes algoritmos y hacerlos intercambiables.
  • Ejemplo: Métodos de ordenamiento.
  • Ventajas: Código más limpio.
  • Desventajas: Puede haber sobrecarga si no se gestiona bien.
  • Usos: Algoritmos personalizables, IA, navegación.
  • Patrones relacionados: State, Template Method.
Template Method
  • Definición: Define la estructura de un algoritmo, delegando pasos específicos a las subclases.
  • Usos: Frameworks, librerías reutilizables.
  • Ventajas: Reutilización.
  • Desventajas: Menor flexibilidad si los pasos están demasiado definidos.
  • Patrones relacionados: Factory Method, Strategy.
Visitor
  • Definición: Permite definir nuevas operaciones sobre una estructura de objetos sin modificar sus clases.
  • Usos: Compiladores, análisis sintáctico.
  • Desventajas: Rígido ante nuevas estructuras.
  • Patrones relacionados: Composite, Interpreter.
